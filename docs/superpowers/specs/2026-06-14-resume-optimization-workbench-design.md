# 推荐结果后简历优化工作台设计

## 目标

在结果页的岗位推荐结果之后，重新设计“简历优化”区域。新的布局应按腾讯校园招聘 skill 的简历优化方式组织信息：先说明简历当前依据，再给出可以立刻执行的修改动作。

本设计采用“左诊断、右行动”的混合工作台结构。它不是独立简历编辑器，也不替代岗位推荐主结果，而是在用户看到推荐岗位后，回答一个更具体的问题：为了这些岗位，简历下一步应该怎么改。

## 腾讯 Skill 输出映射

腾讯校园招聘 skill 的简历优化逻辑包含以下稳定结构：

- 先确认任务类型：诊断现有简历、从零写，或针对特定岗位优化。
- 先肯定亮点，再给具体可执行建议。
- 使用 7 项规则检查作为证据层：教育背景、项目经历、技能列表、STAR 结构、量化成果、表达优化、篇幅。
- 重点提示 STAR、量化成果、弱表达和岗位关键词覆盖。
- 针对岗位方向调整侧重点，例如技术、产品、游戏、设计、市场或职能方向。
- 保持简历正直，只在真实经历基础上优化表达，不编造、不夸大。
- 涉及学校、学历或院校标签时保持中性事实表达，不做价值判断。

UI 需要把这些内容从“长建议列表”变成可扫描的工作台：左侧负责建立可信依据，右侧负责推动用户修改。

## 页面位置

该模块出现在结果页中的“简历优化”区域，承接推荐结果。

推荐位置：

1. 用户完成简历分析后进入结果页。
2. 推荐概览展示岗位列表和当前选中岗位。
3. “简历优化”区域根据当前推荐岗位展示工作台。

顶部文案应明确承接岗位推荐，例如：

`基于当前推荐岗位，简历最该补的是岗位证据、量化结果和个人动作。`

## 桌面端布局

主区域采用两列：

- 左列：诊断依据，占约 40% 宽度。
- 右列：立即修改，占约 60% 宽度。

### 左列：诊断依据

左列帮助用户理解为什么要改。

包含内容：

- 分数与通过数：例如 `72/100 · 5/7 通过`。
- 总体判断：一句话说明当前简历是否具备岗位匹配基础。
- 7 项检查：教育背景、项目经历、技能列表、STAR 结构、量化成果、表达优化、篇幅。
- 亮点可保留：2-3 条真实优势。

7 项检查应使用紧凑状态列表或小表格，不做大面积表格。状态颜色保持克制：

- 通过：绿色。
- 建议改进：琥珀色。
- 不足：红色仅用于缺少关键基础信息。

### 右列：立即修改

右列负责让用户知道现在要做什么。

包含内容：

- 优先改：3 条，按对初筛判断影响排序。
- 岗位定制：围绕当前选中推荐岗位展示关键词、经历侧重和需要补充的证据。
- 改写示例：用“原表达问题 -> 推荐表达”的方式展示。
- 下一步动作：3-5 条可以直接照做的修改动作。

改写示例不能编造未在简历中出现的经历、公司、奖项、项目或数据。没有足够原文证据时，应展示“建议补充的信息类型”，而不是生成具体事实。

## 岗位联动

默认使用当前选中的推荐岗位驱动右列内容。

用户切换推荐岗位时：

- 左列全局诊断保持稳定。
- 右列的岗位定制、关键词、证据补充和改写示例随岗位切换。
- 优先改列表可以保留全局排序，但与岗位相关的问题应优先展示。

如果没有选中岗位，使用推荐列表第一项作为默认岗位。

## 移动端布局

移动端改为上下结构，优先展示行动信息：

1. 分数、通过数和总体判断。
2. 优先改。
3. 改写示例。
4. 岗位定制动作。
5. 7 项检查。
6. 亮点和正直边界。

7 项检查在移动端使用状态列表，不横向滚动。文本需要在窄屏自然换行，按钮、标签和状态徽标不能挤压正文。

## 数据需求

继续复用现有结构化数据：

- `tencentCoaching.resumeAudit`：全局诊断、7 项检查、亮点、优先问题、下一步动作、正直提示。
- `matches[].resumeActions`：岗位级修改动作。
- `matches[].rewriteExample`：岗位级改写示例。
- `matches[].missingKeywords`：岗位缺失关键词。
- `matches[].jdInterpretation.resumeFocus` 或推荐解释中的简历侧重点：岗位定制依据。

如果需要增强“原表达问题 -> 推荐表达”，可在前端先用现有字段做轻量展示：

- 原表达问题：来自 `resumeAudit.prioritizedIssues[].evidence` 或 `matches[].risks`。
- 推荐表达：来自 `matches[].rewriteExample`。

第一版不新增模型调用。

## 兜底行为

当 `resumeAudit` 缺失时：

- 继续展示旧版 `resumeReview` 的亮点、问题、动作和原则。
- 不伪造 7 项检查结果。
- 右列仍可展示选中岗位的 `resumeActions`、`missingKeywords` 和 `rewriteExample`。

当岗位级改写示例缺失时：

- 展示可执行修改动作。
- 提示用户先补充真实项目背景、个人动作和结果证据。

当简历文本过短或基础信息不足时：

- 首要动作应提示先补全教育背景、项目经历、技能列表和目标方向。
- 不展示看似完整的岗位定制改写。

## 视觉要求

- 整体应像结果页里的工作台模块，不像营销页或独立报告页。
- 避免卡片套卡片；两列可以是同一模块内的分区。
- 使用紧凑标题和清晰状态，不使用夸张大字。
- “诊断依据”和“立即修改”要形成明显层级。
- 正直边界用轻量提示，不做沉重警告。
- 保持现有页面风格，但让简历优化区域比普通列表更有结构。

## 测试

需要覆盖：

- 有 `resumeAudit` 时渲染两列工作台结构。
- 显示分数、通过数、7 项检查、优先改和正直提示。
- 显示当前选中岗位的关键词、岗位动作和改写示例。
- 切换推荐岗位后，岗位定制区域随之更新。
- 没有 `resumeAudit` 时保留旧版简历审阅兜底。
- 移动端布局不出现横向溢出。

## 非目标

- 不新增独立简历编辑器。
- 不重做整个结果页。
- 不新增模型调用。
- 不展示通过率、录取率或内部筛选权重。
- 不对学校、学历或院校层级做价值判断。
